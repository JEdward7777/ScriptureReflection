# Codex Editor Edit History — Technical Reference

## Overview

The Codex editor uses an **edit history** system to track all changes to cell content. This history is critical for the **merge resolution** system, which determines which content is authoritative when two versions of a file conflict (e.g., during git sync). If you inject content into `.codex` or `.source` files without adding proper edit history entries, **the Codex editor's merge resolver will drop your content** because it appears to be from an older state.

This document explains the edit history structure, how the merge resolver uses it, and provides sample code for any project that needs to programmatically inject content into Codex files.

---

## The Problem

When content is injected into an existing Codex cell without an accompanying edit history entry, the merge resolution system (`resolveCodexCustomMerge`) compares the edit histories from both sides of a merge. The function `getCellValueData()` in `sharedUtils/index.ts` searches for the **latest edit** where:

1. `editMap` equals `["value"]` (indicating a cell value change)
2. `value` matches the current cell content

If no matching edit entry exists with a recent timestamp, the injected content is treated as stale and gets **reverted to the version that does have a matching edit history entry**.

This is similar to the "file has been unexpectedly modified" errors seen in other editor tools ([github.com](https://github.com/anthropics/claude-code/issues/10882), [github.com](https://github.com/sst/opencode/issues/5840)) — the system uses timestamps/hashes to detect whether content was properly authored through the expected channels.

---

## Edit History Data Structure

Each cell in a `.codex` or `.source` file has a `metadata.edits` array. Each entry in this array is an `EditHistory` object with the following structure:

```json
{
  "editMap": ["value"],
  "value": "The cell content after this edit",
  "timestamp": 1710940800000,
  "type": "user-edit",
  "author": "username",
  "validatedBy": []
}
```

### Field Reference

| Field | Type | Description |
|-------|------|-------------|
| `editMap` | `string[]` | Path identifying what was edited. For cell content changes, this is always `["value"]`. Other paths exist for metadata fields (e.g., `["metadata", "cellLabel"]`, `["metadata", "data", "startTime"]`). |
| `value` | `string \| number \| boolean` | The new value after this edit. For cell content edits, this is the HTML string content of the cell. |
| `timestamp` | `number` | Unix timestamp in **milliseconds** (matching JavaScript's `Date.now()`). This is the key field the merge resolver uses to determine which edit is most recent. |
| `type` | `string` | One of the `EditType` enum values (see below). |
| `author` | `string` | The username/identifier of who made the edit. Use a descriptive name like `"easy_draft"` or your tool's name. |
| `validatedBy` | `ValidationEntry[]` | Array of validation entries. For programmatic injection, use an empty array `[]`. |

### EditType Values

These are defined in the codex-editor's `types/enums.ts`:

| Value | When to Use |
|-------|-------------|
| `"user-edit"` | Content was intentionally changed by a user or tool. **Use this when updating existing cells.** |
| `"llm-edit"` | Content was edited by an LLM. |
| `"llm-generation"` | Content was generated by an LLM. |
| `"initial-import"` | Content was imported for the first time. **Use this for new cells or to record the original value before overwriting.** |
| `"merge"` | Content resulted from a merge operation. |
| `"migration"` | Content was changed by a data migration. |

### ValidationEntry Structure

```json
{
  "username": "some_user",
  "creationTimestamp": 1710940800000,
  "updatedTimestamp": 1710940800000,
  "isDeleted": false
}
```

For programmatic injection, you typically leave `validatedBy` as an empty array `[]`.

---

## How the Merge Resolver Uses Edit History

The merge resolution happens in `resolveCodexCustomMerge()` (in `resolvers.ts`). Here's the key logic:

1. **For each cell that exists in both versions**, it calls `resolveMetadataConflictsUsingEditHistory()`.
2. This function **combines all edits from both versions** and groups them by `editMap` path.
3. For each path (e.g., `"value"`), it finds the **most recent edit** (highest `timestamp`).
4. The most recent edit's `value` is applied to the merged cell.

**Tie-breaking rules** (when timestamps are equal):
- `"migration"` type wins over all others
- `"user-edit"` wins over `"initial-import"`
- `"initial-import"` loses to everything else

This means: **if your injected content has no edit history entry, or has an entry with an older timestamp than the existing content's edit history, your content will be dropped.**

## Sample Code

### Python — Creating an Edit History Entry

```python
import time

def create_edit_history_entry(value, timestamp_ms=None, edit_type="user-edit",
                               author="my_tool"):
    """
    Create an edit history entry compatible with the Codex editor.

    Args:
        value: The cell content string after this edit
        timestamp_ms: Unix timestamp in milliseconds. If None, uses current time.
        edit_type: One of: "user-edit", "initial-import", "llm-generation", etc.
        author: Identifier for who/what made this edit

    Returns:
        Dict matching the Codex EditHistory structure
    """
    if timestamp_ms is None:
        timestamp_ms = int(time.time() * 1000)

    return {
        'editMap': ['value'],
        'value': value,
        'timestamp': timestamp_ms,
        'type': edit_type,
        'author': author,
        'validatedBy': [],
    }
```

### Python — Updating an Existing Cell

When updating an existing cell's value, you need to:

1. **Check if the content is actually changing** — skip if identical
2. **Record the old value** as an `"initial-import"` if the cell has no prior edit history
3. **Add a new edit entry** for the new value with a current timestamp

```python
import time
import json

def inject_content_into_cell(cell, new_content, author="my_tool"):
    """
    Update a cell's value and add proper edit history.

    Args:
        cell: The cell dict from the codex JSON structure
        new_content: The new content string to set
        author: Tool identifier for the edit history
    """
    old_content = cell.get('value', '')

    # Don't create edit history if content isn't changing
    if old_content == new_content:
        return

    # Update the cell value
    cell['value'] = new_content

    # Ensure metadata.edits exists
    metadata = cell.setdefault('metadata', {})
    edits = metadata.setdefault('edits', [])

    timestamp_ms = int(time.time() * 1000)

    # If this cell had content but no edit history, record the old value first.
    # This matches the pattern in codexDocument.ts updateCellContent() which
    # creates a backdated initial-import entry for the previous value.
    if len(edits) == 0 and old_content and old_content.strip():
        edits.append({
            'editMap': ['value'],
            'value': old_content,
            'timestamp': timestamp_ms - 1000,  # 1 second before the new edit
            'type': 'initial-import',
            'author': author,
            'validatedBy': [],
        })

    # Add the new edit entry
    edits.append({
        'editMap': ['value'],
        'value': new_content,
        'timestamp': timestamp_ms,
        'type': 'user-edit',
        'author': author,
        'validatedBy': [],
    })
```

### Python — Creating a New Cell with Edit History

When creating a brand new cell, include an `"initial-import"` edit entry:

```python
def create_new_cell(cell_id, content, global_ref, cell_label, author="my_tool"):
    """
    Create a new codex cell with proper edit history.
    """
    timestamp_ms = int(time.time() * 1000)

    return {
        'kind': 2,
        'languageId': 'html',
        'value': content,
        'metadata': {
            'type': 'text',
            'id': cell_id,
            'edits': [{
                'editMap': ['value'],
                'value': content,
                'timestamp': timestamp_ms,
                'type': 'initial-import',
                'author': author,
                'validatedBy': [],
            }],
            'data': {
                'globalReferences': [global_ref]
            },
            'cellLabel': cell_label,
        }
    }
```

### JavaScript/TypeScript — Equivalent Code

If your injection tool is written in TypeScript/JavaScript:

```typescript
interface EditHistory {
  editMap: string[];
  value: string | number | boolean;
  timestamp: number;
  type: string;
  author: string;
  validatedBy: ValidationEntry[];
}

interface ValidationEntry {
  username: string;
  creationTimestamp: number;
  updatedTimestamp: number;
  isDeleted: boolean;
}

function createEditHistoryEntry(
  value: string,
  editType: string = "user-edit",
  author: string = "my_tool"
): EditHistory {
  return {
    editMap: ["value"],
    value,
    timestamp: Date.now(),
    type: editType,
    author,
    validatedBy: [],
  };
}

function injectContentIntoCell(
  cell: any,
  newContent: string,
  author: string = "my_tool"
): void {
  const oldContent = cell.value || "";

  // Skip if content isn't changing
  if (oldContent === newContent) return;

  cell.value = newContent;

  if (!cell.metadata) cell.metadata = {};
  if (!cell.metadata.edits) cell.metadata.edits = [];

  const now = Date.now();

  // Record old value if no prior history exists
  if (cell.metadata.edits.length === 0 && oldContent.trim()) {
    cell.metadata.edits.push({
      editMap: ["value"],
      value: oldContent,
      timestamp: now - 1000,
      type: "initial-import",
      author,
      validatedBy: [],
    });
  }

  // Add new edit entry
  cell.metadata.edits.push({
    editMap: ["value"],
    value: newContent,
    timestamp: now,
    type: "user-edit",
    author,
    validatedBy: [],
  });
}
```

## Complete Example: Reading, Modifying, and Writing a Codex File

```python
import json
import time

def inject_into_codex_file(filepath, cell_id, new_content, author="my_tool"):
    """
    Find a cell by ID in a codex file and update its content with proper
    edit history so the merge resolver won't revert it.
    """
    with open(filepath, 'r', encoding='utf-8') as f:
        notebook = json.load(f)

    cells = notebook.get('cells', [])

    for cell in cells:
        if cell.get('metadata', {}).get('id') == cell_id:
            old_content = cell.get('value', '')

            # Skip if nothing is changing
            if old_content == new_content:
                print(f"Cell {cell_id}: content unchanged, skipping")
                return

            # Update value
            cell['value'] = new_content

            # Add edit history
            metadata = cell.setdefault('metadata', {})
            edits = metadata.setdefault('edits', [])
            timestamp_ms = int(time.time() * 1000)

            if len(edits) == 0 and old_content and old_content.strip():
                edits.append({
                    'editMap': ['value'],
                    'value': old_content,
                    'timestamp': timestamp_ms - 1000,
                    'type': 'initial-import',
                    'author': author,
                    'validatedBy': [],
                })

            edits.append({
                'editMap': ['value'],
                'value': new_content,
                'timestamp': timestamp_ms,
                'type': 'user-edit',
                'author': author,
                'validatedBy': [],
            })

            print(f"Cell {cell_id}: updated with edit history (ts={timestamp_ms})")
            break
    else:
        print(f"Cell {cell_id}: not found in {filepath}")
        return

    with open(filepath, 'w', encoding='utf-8') as f:
        json.dump(notebook, f, ensure_ascii=False, indent=2)
        f.write('\n')
```

## Key Rules to Remember

1. **Always use millisecond timestamps** — `int(time.time() * 1000)` in Python, `Date.now()` in JavaScript.

2. **Always set `editMap` to `["value"]`** for cell content changes. The merge resolver specifically looks for this path.

3. **The `value` field in the edit entry must exactly match the cell's `value` field.** The `getCellValueData()` function compares them with strict equality.

4. **Don't create edit history entries when content isn't changing.** This avoids unnecessary history bloat and potential confusion in the merge resolver.

5. **When overwriting a cell that has no prior edit history**, create a backdated `"initial-import"` entry for the old value first (timestamp - 1000ms). This preserves the full history chain.

6. **Use `"initial-import"` for new cells** and `"user-edit"` for updates to existing cells. The merge resolver gives `"user-edit"` priority over `"initial-import"` when timestamps are equal.

7. **The `validatedBy` array can be empty `[]`** for programmatic injection. Validation is a separate concern handled by the Codex editor UI.

8. **Use a descriptive `author` string** (e.g., `"easy_draft"`, `"my_import_tool"`) so edits can be traced back to their source.

## File Structure Reference

A Codex notebook file (`.codex` or `.source`) has this top-level structure:

```json
{
  "cells": [
    {
      "kind": 2,
      "languageId": "html",
      "value": "<p>Cell content here</p>",
      "metadata": {
        "type": "text",
        "id": "cell-uuid-here",
        "edits": [
          {
            "editMap": ["value"],
            "value": "<p>Cell content here</p>",
            "timestamp": 1710940800000,
            "type": "user-edit",
            "author": "easy_draft",
            "validatedBy": []
          }
        ],
        "data": {
          "globalReferences": ["GEN 1:1"]
        },
        "cellLabel": "1"
      }
    }
  ],
  "metadata": {
    "id": "Genesis",
    "originalName": "Genesis",
```
